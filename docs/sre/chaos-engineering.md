# 🌀 カオスエンジニアリング (Chaos Engineering) 実験計画書

本ドキュメントは、システム（Route 53 ➔ CloudFront ➔ ALB ➔ ECS Fargate ➔ RDS Aurora / ElastiCache Redis）の回復力（Resiliency）を実証するため、本番環境およびステージング環境において意図的に障害を注入し、自動復旧メカニズムが期待通り機能するかを検証するためのカオスエンジニアリング実験計画を定義します。

---

## 1. カオスエンジニアリングの原則 ＆ ガードレール

本システムにおける実験は、以下の原則とガードレールに基づいて安全に実行されます。

### 1) 定常状態 (Steady State) の定義
実験の開始前および最中において、システムが健全であることを判定する指標として [SLO定義書](file:///c:/Git/learning-ts-concepts/docs/sre/slo-sli.md) の数値を常時モニタリングします。
*   **定常状態の基準**:
    *   API可用性 SLI: **99.9% 以上** (HTTP 5xx の発生率が 0.1% 未満であること)
    *   APIレスポンス速度 (p95) SLI: **500ms 以下**

### 2) 非常停止スイッチ (Emergency Rollback / Blast Radius Control)
カオス実験が想定外の致命的障害を引き起こし、SLOバジェットを急速に消費し始めた場合は、即座に実験を中止し回復措置（ロールバック）を実行します。
*   **AWS FIS 停止条件 (Stop Conditions)**:
    *   AWS CloudWatch Alarm / Datadog Monitor で `HTTPCode_Target_5XX_Count > 50` を検知した際、実行中の **AWS FIS (Fault Injection Service)** の実験テンプレートを自動停止（ロールバック）させます。

---

## 2. カオス実験テンプレート (Chaos Experiment Templates)

### 🧪 実験 1: データベース (Aurora) のフェイルオーバー耐性検証

* **目的**: データベースのプライマリノードが被災した際、自動フェイルオーバーと RDS Proxy によって、アプリケーションがエラーを発生させずに通信を継続できるかを実証する。
* **仮説**:
  > 「DBのフェイルオーバー中（通常30〜60秒）も、コンテナとDBの間に介在する RDS Proxy がコネクションをキューイング（保持）して接続を維持するため、ユーザーにはHTTP 5xx エラーが返ることなく、わずかな遅延のみでリクエストが処理されるはずである。」
* **障害注入方法**:
  AWS CLI または AWS FIS を使用し、Aurora DB クラスターのフェイルオーバーを強制実行する。
  ```bash
  aws rds failover-db-cluster \
    --db-cluster-identifier prod-database-cluster \
    --region ap-northeast-1
  ```
* **検証成功基準 (Success Criteria)**:
  1.  ALB の `HTTPCode_Target_5XX_Count` が **0 件** であること。
  2.  Datadog APM トレースにおいて、DBフェイルオーバー中の API レスポンス遅延が一時的に上昇するものの、タイムアウト（Fargate側の5秒制限）に達せず正常応答すること。

---

### 🧪 実験 2: アベイラビリティゾーン (AZ) 障害の再現

* **目的**: 特定データセンター（AZ）の物理的な電源喪失やネットワーク分断が発生した際、マルチAZ設計の ECS および ALB が自動で影響を回避し、サービスを継続できるかを実証する。
* **仮説**:
  > 「3つのAZに分散配置された ECS Fargate のうち、1つのAZが完全に機能不全に陥っても、ALBのターゲットグループのヘルスチェックによって被災したタスクが自動で切り離され、稼働中の他AZのタスク（Min 2台）へトラフィックが即座に流れるため、サービス全体の可用性に影響を与えないはずである。」
* **障害注入方法**:
  **AWS FIS (Fault Injection Service)** を使用し、特定のサブネット（例: `ap-northeast-1a` 用の Private Subnet）に対するセキュリティグループルールを変更し、アウトバウンド/インバウンド通信を完全にドロップさせる。
* **検証成功基準 (Success Criteria)**:
  1.  可用性 SLO が **99.9%** を維持していること。
  2.  影響を受けた AZ の ECS タスクが ALB によって `Unhealthy` と判定され、数秒以内にルーティングから切り離されること。
  3.  ECS の Auto Scaling が発火し、被災していない AZ-b または AZ-c に代替タスクが自動プロビジョニングされること。

---

### 🧪 実験 3: Redis キャッシュ層のネットワーク切断（フォールバック検証）

* **目的**: キャッシュ層（ElastiCache Redis）が完全に停止（被災）した際、アプリケーションがハングアップせず、自動的に DB 直接参照へフォールバックして稼働を継続できるか（Graceful Degradation）を検証する。
* **仮説**:
  > 「Redis への接続が切断された場合、アプリケーションのキャッシュクライアントが接続エラーを検知してタイムアウト（秒単位）し、自動的にDBから直接データを引くルートへ切り替えてレスポンスを返すはずである（DB負荷は上がるが、サービスは停止しない）。」
* **障害注入方法**:
  Redis セキュリティグループのインバウンドルールを一時的に変更し、ECS セキュリティグループからのアクセスを遮断（拒否）する。
* **検証成功基準 (Success Criteria)**:
  1.  API エラー率が **0%** を維持していること（Redis接続エラーがアプリケーション内部でキャッチされ、500エラーとして表面化しないこと）。
  2.  API レスポンス速度 (p95) が一時的に増加（キャッシュミスによるDBアクセス増）するものの、サービスが継続していること。
  3.  監視ダッシュボードで Aurora の `CPUUtilization` および `DatabaseConnections` が一時的に上昇することを確認し、フォールバックの動作を実証する。

---

## 3. 実験の実行ワークフローと評価

カオス実験は、ステージング環境 (`stg`) で十分にテストされ、安全性が確認された実験テンプレートのみを、深夜などの低トラフィック帯に本番環境 (`prod`) へ適用します。

```
 [定常状態の計測] (Datadog SLO監視の確認)
       │
       ▼
 [AWS FIS 実験テンプレートの実行] (15分間の障害注入)
       │
       ├─► 異常検知（Stop Condition発火）➔ 即座に自動ロールバック・復旧
       │
       ▼ (実験完了)
 [ポスト実験分析 (Game Day Report)]
   - 仮説は正しかったか？
   - 復旧に想定以上の時間がかかったボトルネックはどこか？
   - 改善項目をバックログへ起票 (チケット化)
```

実験結果はすべて「SRE Game Day Report」として起票・ドキュメント化され、インフラ構成のさらなる耐久性向上（リトライポリシー、タイムアウト値のチューニング等）へと還元されます。
