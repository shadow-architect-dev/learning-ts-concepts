# 🌋 災害復旧 (Disaster Recovery: DR) 方針書

本ドキュメントは、広域災害（大規模地震等）やリージョン障害、壊滅的なデータ破損が発生した際に、本アプリケーションシステム（Route 53 + CloudFront + ALB + ECS Fargate + RDS Aurora + ElastiCache Redis）のデータ資産を守り、迅速にシステムを復旧させるための災害復旧方針を定義します。

---

## 1. DR 目標指標 (RTO / RPO)

本サービスでは、事業継続計画（BCP）に基づき、以下のサービスリカバリ指標をターゲットに設定します。

```
              ┌── 災害発生 (Disaster Event)
              │
◄─────────────┼─────────────────────────────►
  RPO: 1時間以内   │        RTO: 4時間以内
 (許容データ損失)  │       (システム復旧時間)
```

* **RTO (Recovery Time Objective: 目標復旧時間)**: **4時間以内**
  * 災害発生（被災またはリージョン全面停止宣言）から、代替リージョンでユーザーアクセスを受け入れ可能にするまでの目標時間。
* **RPO (Recovery Point Objective: 目標復旧時点)**: **1時間以内**
  * 災害発生時に失われるデータの最大許容時間。DBのデータおよび静的アセットは、常に過去1時間以内の状態まで復元できるようにバックアップ設計を行います。

---

## 2. バックアップ ＆ レプリケーション戦略 (常時稼働)

RPO 1時間をクリアするため、以下の多層データバックアップ体制を IaC (CDK) にて自動化・常時稼働させています。

### 1) Amazon Aurora DB (データ層)
* **リージョン内冗長化**: 
  データベースストレージは、東京リージョンの3つのアベイラビリティゾーン（AZ）にわたって自動で6冗長（各AZに2コピー）に保存されます。単一データセンターの被災ではデータロスは発生しません。
* **日次自動スナップショット**:
  毎日深夜にフルスナップショットを取得し、35日間保持。
* **クロスリージョンコピー (KMS暗号化付き)**:
  本番（`prod`）環境では、スナップショット完了後、自動的に大阪リージョン（`ap-northeast-3`）へコピー。コピー時の転送データは大阪リージョン側の KMS CMK で再暗号化され、安全に保管されます。
* **ポイントインタイムリカバリ (PITR)**:
  トランザクションログを継続的に S3 へバックアップしており、過去35日間の任意の時点（秒単位）にDBの状態を巻き戻すことが可能です。

### 2) S3 静的アセットバケット (ストレージ層)
* **クロスリージョンレプリケーション (CRR)**:
  アセット専用の S3 バケット ([storage.ts](file:///c:/Git/learning-ts-concepts/infra/lib/constructs/storage.ts)) に対し、バケットのバージョニングを有効化。
  本番環境では、アップロードされたすべてのファイルが、大阪リージョンに構築されたバックアップ用 S3 バケットへリアルタイムで非同期レプリケーション（CRR）されます。これにより、S3オブジェクトの RPO はほぼ 0 に抑えられます。

## 3. 被災シナリオ別の復旧手順

### シナリオ A: 単一アベイラビリティゾーン (AZ) の物理障害
* **状況**: 東京リージョン内の一つのデータセンター群で電源喪失や火災が発生。
* **復旧アクション (自動化)**:
  * **ECS Fargate**: タスクは元々異なる AZ（3 AZ）に自動分散配置されているため、被災した AZ のタスクが停止すると、ECS サービスが自動で別の健全な AZ へ代替タスクを即時プロビジョニングします。
  * **Aurora Serverless**: 自動的に健全な AZ にあるリードレプリカがプライマリ（ライター）へと昇格（フェイルオーバー）します（復旧時間：約30秒〜1分）。
  * **ElastiCache (Redis)**: 自動フェイルオーバーにより、数秒でレプリカノードがプライマリへ昇格します。
  * **ユーザー影響**: 数十秒の一時的な接続エラーを除き、システムは完全に稼働を継続します（SREの手動介入不要）。

---

### シナリオ B: 東京リージョン（プライマリ）全体の崩壊・全面停止
* **状況**: 地震等の激甚災害により、東京リージョン全体のアベイラビリティが喪失。
* **復旧アクション (手動 DR 発動手順 / SREチーム実行)**:

```
[東京リージョン被災]
        │
        ▼ (SRE判断でDR発動)
[大阪リージョンへCDKデプロイ] ── (cdk deploy --context env=prod-dr --region ap-northeast-3)
        │
        ▼
[大阪のS3アセット・KMSキーが活性化]
        │
        ▼
[直近のAuroraスナップショットからDB復元] ── (RPO: 1時間以内)
        │
        ▼
[Route 53 DNS 切り替え] ── (DNSレコードを大阪のCloudFrontへ)
        │
        ▼
[サービス再開 (RTO: 4時間以内クリア)]
```

#### Step 1. 大阪リージョンでのインフラプロビジョニング
SREは、DR用環境設定（`envName=prod-dr`）およびリージョンに `ap-northeast-3` を指定し、大阪リージョンへ CDK スタックをデプロイします。
```powershell
npx cdk deploy ThreeTierStack-prod-dr --region ap-northeast-3 -c env=prod-dr
```
※ VPC、ALB、ECS クラスター、Redis、S3バケット等のステートレスなインフラが大阪リージョンに即時展開されます。

#### Step 2. データベースの復旧
1. コピー済みの最新の Aurora 自動スナップショット（または任意の時点の PITR データ）を特定。
2. 大阪リージョンに新規デプロイされた Aurora DB クラスターに対して、スナップショットからのリストア（復元）を AWS CLI / コンソールから実行します。
3. リストア完了後、Secrets Manager のシークレット値（接続エンドポイント名）を大阪リージョンのDBホスト名に更新します。

#### Step 3. トラフィックのルーティング変更 (DNS フェイルオーバー)
Route 53 Hosted Zone の DNS レコード（エイリアスレコード）を、大阪リージョンにデプロイされた新しい CloudFront Distribution のドメイン名へ書き換えます。
DNSのTTLキャッシュが伝播すると、世界中のユーザーからのトラフィックが自動的に大阪リージョンへと切り替わります（RTO目標時間内での復旧完了）。

## 4. DR 訓練計画 (DR Drill)

本方針書の有効性を担保し、インシデントコマンダーのスキル維持を図るため、以下のスケジュールで定期的な災害復旧訓練を実施します。

* **実施頻度**: 年1回
* **訓練内容**:
  * 東京リージョンから大阪リージョンへの IaC (CDK) による全スタックのテスト展開。
  * バックアップスナップショットからの Aurora テスト復元と、アプリコンテナの疎通確認。
  * 切り替え手順書のアップデート、およびデプロイ所要時間（RTO）が想定内に収まっているかの測定。
  * 訓練終了後は必ず「DR訓練ポストモーテム」を作成し、手順の自動化余地や課題を次回スプリントのバックログに積み上げます。
