* **インシデント管理番号**: INC-20260615-01
* **ステータス**: 完了 (Resolved)
* **重要度 (Severity)**: Severity 2 (部分的なサービス機能低下)
* **影響時間**: 2026-06-15 14:00 - 14:15 JST (計15分間)
* **対象環境**: 本番環境 (`prod`)
* **インシデントコマンダー (SRE)**: @sre-oncall-john
* **主な影響**: 
  * APIへのリクエストに対し、一時的にHTTP 500（内部サーバーエラー）が急増。
  * 可用性SLO（サービスレベル目標）の目標値「99.9%」に対し、直近30日のエラーバジェットを約 12% 消費。
  * 影響を受けたユーザー数：全アクティブセッションの約 8.5%

## 1. インシデント概要

2026年6月15日 14:00 JST、本番環境のデータベース（Aurora Serverless v2）に対する認証情報の定期自動ローテーション（Secrets Managerによる30日周期実行）が走った際、実行中のアプリケーションコンテナ（ECS Fargate）からDBへの接続エラーが突発的に発生しました。
これにより、15分間にわたりAPIの応答成功率が低下しました。SREチームによるECSタスクの強制再起動（キャッシュクリア）によって14:15にシステムは完全復旧しました。

## 3. 根本原因分析 (5-Whys)

SREチームは以下の「5-Whys（5つのなぜ）」分析を用いて、根本原因を突き止めました。

1. **なぜコンテナからのDB接続がエラー（Access Denied）になったのか？**
   * ➔ コンテナのメモリ内にキャッシュされていた古いDBパスワードを使って、新パスワード変更後のDBへ接続しようとしたため。
2. **なぜ古いパスワードを使い続けたのか？**
   * ➔ ECSコンテナは起動時にのみ Secrets Manager から環境変数・シークレット値をロードするため、実行中にパスワードが変わっても追従する仕組み（リトライ時の Secrets Manager 再参照ロジック）をアプリ内に持っていなかったため。
3. **なぜ無停止ローテーションを担う「RDS Proxy」が機能しなかったのか？**
   * ➔ アプリケーションコンテナが RDS Proxy のエンドポイントではなく、DB クラスターの生のエンドポイント（Writerホスト）に直接接続していたため。
4. **なぜコンテナが直接 DB クラスターを向いてしまっていたのか？**
   * ➔ CDKコード内の `stack.ts` にて、`dbHost` 変数が Proxy の有無に関わらず常に DB クラスターを向くように設定された「デグレーション（構成先戻り）」が発生していたため。
5. **なぜデグレーションの発生にデプロイ前に気づけなかったのか？**
   * ➔ CDKのインフラアサーションテスト（Jest）において、「本番環境（`prod`）で `DB_HOST` 環境変数が RDS Proxy のエンドポイントを指していること」を確認するテストコードが欠落していたため。

## 4. 再発防止策 (Preventative Actions)

SREチームでは、二度と同様の認証不整合を発生させないために以下の恒久対策を実施します。

### 短期対策 (即時実施)
* **[ACTION-01] CDKテストの強化 (完了)**:
  * Jest テストコード（[stack.test.ts](file:///c:/Git/learning-ts-concepts/infra/test/stack.test.ts)）において、`stg` / `prod` スタックのテストで `DB_HOST` の値が `DBProxy` のエンドポイント属性（`Endpoint.Address`）を正しく参照していることをアサーション検証するチェックを実装しました。これにより、今後のコード変更時の先戻りを自動ブロックします。
* **[ACTION-02] アプリの接続文字列パラメータの修正 (完了)**:
  * アプリが必ず RDS Proxy のエンドポイントを経由して通信するように環境変数へのバインド定義を修正しました。

### 長期対策 (次期スプリント)
* **[ACTION-03] アプリケーション接続リトライロジック of 改善 (未完了/チケット: SRE-128)**:
  * アプリケーション側の DB 接続プールライブラリに対して、`Access Denied` エラーを検知した際には、バックオフを取りつつ即座に Secrets Manager SDK を叩いてキャッシュをクリアし、最新パスワードを再取得して再試行する自動回復ロジックを実装します。
* **[ACTION-04] RDS Proxy 死活監視アラート of 構築 (未完了/チケット: SRE-129)**:
  * RDS Proxy の接続プール枯渇（`DatabaseConnections`）およびヘルスチェック不調を検知する Datadog Monitor を CDKTF（`monitoring/`）に定義し、DB本体だけでなく中継層の監視を強化します。

## 5. 学びと評価 (SREチームより)

今回のインシデントにおいて、Datadog アラートの発報からオンコールの調査開始まで「3分」、暫定対処の実施まで「7分」と、非常に高い即応性を発揮できました。
しかし、自動ローテーションという「正常系プロセス」が障害のトリガーとなったことは、テスト自動化とインフラ検証の重要性を改めて示す結果となりました。
今回の教訓をもとに、IaCのテストに「本番固有の設定値（RDS Proxy）」のアサーションを組み込むことで、システムの「静的検証性（Static Resiliency）」をさらに高めることができました。
チームメンバーの迅速な対応と、冷静な根本原因の分析（Blameless）に感謝します。
