# トラブルシューティング手順書: 静的アセット配信エラー (CloudFront / S3 アラート)

本ドキュメントは、CloudFront を通じた静的アセット（`/assets/*`）の配信エラー率（5xx / 4xx）が急増し、可用性 SLO（99.99%）を脅かすアラートが発生した際の一次対応手順書（Runbook）です。

---

## 1. エラーパターンの特定

CloudFront または S3 のログ、もしくは Datadog メトリクスを確認し、エラーのステータスコードを特定します。

* **`403 Forbidden`**: S3 バケットへのアクセス権限がない、または OAC の認証失敗。
* **`404 Not Found`**: 指定された静的アセットファイルが S3 内に存在しない。
* **`5xx Server Error`**: CloudFront エッジと S3 オリジン間の接続、または S3 側の障害。

---

## 2. 要因別の切り分け・対応フロー

### A. `403 Forbidden` エラーが発生している場合
CloudFront が S3 バケットからアセットを取得しようとした際に、権限エラーが発生しています。

1. **Origin Access Control (OAC) 設定の確認**:
   * AWS Console の CloudFront ディストリビューション設定を開き、S3 オリジンに対して OAC（Origin Access Control）が設定されているか確認。
2. **S3 バケットポリシーの確認**:
   * 対象の S3 アセットバケットのバケットポリシーを開き、CloudFront サービスプリンシパル (`cloudfront.amazonaws.com`) からの `s3:GetObject` が許可されており、かつ条件 (Condition) に該当の CloudFront ディストリビューション ARN が正しく指定されているか確認。
3. **一次対応**:
   * ポリシー不整合の場合 ➔ `shared-outputs.md` のディストリビューションIDやアカウントIDに誤りがないか確認し、CDK デプロイを再実行してバケットポリシーを自動修正。

### B. `404 Not Found` エラーが発生している場合
ユーザーがリクエストしたアセットが、S3 バケット内に存在しないか、パスが間違っています。

1. **S3 バケット内のオブジェクト確認**:
   * S3 コンソールを開き、該当のアセット（例: `/assets/js/main.js`）がバケット内に実際に配置されているか確認。
2. **CI/CD デプロイログの確認**:
   * GitHub Actions の最新のデプロイジョブを確認し、ビルドされたフロントエンドアセットを S3 バケットへ同期（`aws s3 sync`）するステップが正常に完了していたか確認。
3. **一次対応**:
   * アセットの未配置 ➔ GitHub Actions からビルド＆アセットアップロードステップを手動で再実行。
   * パスの不整合（ハッシュ値の変更など） ➔ アプリケーションコンテナ側の HTML/JS 内で参照しているアセットのファイル名と、S3 にアップロードされたファイル名が一致しているか確認。

### C. `504 Gateway Timeout` または `502 Bad Gateway` の場合

1. **S3 サービスステータスの確認**:
   * AWS Service Health Dashboard を開き、S3 サービス自体に大規模な障害（障害によるエラーレートの上昇）が発生していないか確認。
2. **一時的なネットワーク遅延**:
   * CloudFront から S3 への接続で一時的なパケットロスが発生していないか監視。

---

## 3. インシデントの報告・エスカレーション

復旧作業が 15分 以内に完了しない、あるいは原因が特定できない場合は、速やかに SRE チャネル (`#sre-incident`) に以下の情報を報告してください。

* 発生時刻 (UTC / JST)
* エラーが発生しているアセットの URL (例: `https://xxxx.cloudfront.net/assets/css/style.css`)
* エラーコード (403 / 404 / 503 等)
* S3 バケットポリシーの最新の変更履歴
