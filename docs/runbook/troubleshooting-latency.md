# トラブルシューティング手順書: API レスポンス遅延 (Latency アラート)

本ドキュメントは、API の応答速度が低下し、レイテンシ SLO（p95 < 500ms）を脅かすアラートが発生した際の一次対応手順書（Runbook）です。

---

## 1. ボトルのネックの特定（APMの確認）

Datadog APM (Trace) または AWS X-Ray を使用し、リクエスト処理時間の大半がどこで消費されているかを特定します。

* **データベース処理 (MySQL / RDS)**: クエリ処理に時間がかかっているか。
* **キャッシュ処理 (Redis)**: Redis への接続遅延、またはキャッシュヒット率の低下。
* **コンテナ内アプリロジック**: CPU やメモリの制限に達してプロセスが遅延しているか。

---

## 2. 要因別の切り分け・対応フロー

### A. データベース (RDS Aurora) にボトルネックがある場合

1. **RDS Proxy のキューとコネクション確認**:
   * RDS Proxy の `DatabaseConnectionsCurrentlyEstablished`（確立された接続数）が上限に達していないか確認。
2. **スロークエリの特定**:
   * Datadog Database Monitoring または AWS Performance Insights を開き、CPU やディスク I/O を大量に消費しているスロークエリを特定。
3. **一次対応**:
   * リードレプリカ (Reader) の有効活用 ➔ 参照処理がマスター (Writer) に集中している場合、プログラムの接続先が Reader (DB ProxyのReaderエンドポイント) を向いているか確認。
   * インデックスの不足 ➔ 緊急で必要なインデックス (`CREATE INDEX`) をデータベースに適用し、クエリ走査数を削減。

### B. キャッシュ (ElastiCache Redis) にボトルネックがある場合

1. **キャッシュヒット率とCPU使用率の確認**:
   * Redis の CPU 使用率（特に `EngineCPUUtilization`）が 80% を超えていないか確認（Redisはシングルスレッド動作のため、EngineCPU の高負荷は致命的）。
   * キャッシュヒット率が急激に低下していないか確認。
2. **一次対応**:
   * ホットキー（特定キーへのアクセス集中）の調査 ➔ コマンド `redis-cli --hotkeys` 等でアクセスが集中しているキーを特定し、プログラム側で一時的にローカルメモリキャッシュ等に逃がす。
   * クラスターのスケールアップ ➔ キャッシュ容量自体が不足している場合、CDK パラメータでノードサイズを拡張してデプロイ。

### C. ECS Fargate コンテナの負荷が高い場合

1. **タスクの CPU / メモリ使用率確認**:
   * 各 ECS タスクの CPU/メモリ使用率が 70% のオートスケーリング閾値付近で推移しているか確認。
   * オートスケーリングが正常にトリガーされ、Desired Tasks（希望タスク数）が増加しているか確認。
2. **一次対応**:
   * スケーリングが追いついていない場合 ➔ AWS Console より、ECS サービスの「希望タスク数」を手動で直接増大させてトラフィックを分散（水平スケールアウト）。
   * メモリリークの疑い ➔ 特定のコンテナが時間経過とともにメモリを使い果たしている場合、タスクを順次ローリング再起動し、メモリを解放。

---

## 3. インシデントの報告・エスカレーション

復旧作業が 15分 以内に完了しない、あるいは原因が特定できない場合は、速やかに SRE チャネル (`#sre-incident`) に以下の情報を報告してください。

* 発生時刻 (UTC / JST)
* 遅延が顕著な API パス (例: `/api/v1/checkout`)
* 平均レイテンシ (p95 / p99 のミリ秒値)
* データベースおよびコンテナの CPU 使用率
