# トラブルシューティング手順書: API 可用性低下 (HTTP 5xx アラート)

本ドキュメントは、API のエラーレートが急増し、可用性 SLO（99.9%）を脅かすアラートが発生した際の一次対応手順書（Runbook）です。

---

## 1. 状況の確認と影響範囲の特定

まず、Datadog ダッシュボードまたは AWS コンソールを開き、エラーの発生源とステータスコードを特定します。

1. **Datadog SLO / ALB ダッシュボードの確認**:
   * エラーが「一部の API パス」のみで発生しているか、「システム全体」で発生しているかを確認。
2. **HTTP ステータスコードの特定**:
   * エラーが `502`、`503`、`504` のいずれであるかを確認します。これにより原因を素早く絞り込めます。

---

## 2. ステータスコード別の切り分け・対応フロー

### A. `502 Bad Gateway` (ECSコンテナ / アプリの異常)
ALB から ECS Fargate タスクへの通信、またはタスク自体の応答が途絶えている場合に発生します。

1. **ECS タスクの稼働状態確認**:
   * AWS Console の ECS サービスを開き、実行中のタスク数 (Running Tasks) が希望数 (Desired Tasks) を維持しているか確認。
   * タスクの再起動（CrashLoopBackOff）が繰り返されている場合、直前のリリースコードにバグがある可能性が高いです。
2. **コンテナログの調査**:
   * CloudWatch Logs (`/ecs/env/AppContainer`) または Datadog APM を開き、例外エラー (Exception) や Out of Memory (OOM) ログが出ていないか調査。
3. **一次対応**:
   * 直前のデプロイが原因の場合 ➔ GitHub Actions から以前の安定バージョンへロールバックを実行。
   * 一時的なコンテナのハングアップの場合 ➔ ECS サービスを「新しいデプロイの強制」でタスクをローリング再起動。

### B. `503 Service Unavailable` (WAFブロック / サービス全滅)
ALB へのリクエストが手前で遮断されているか、背後の ECS タスクが 1 台も起動していない場合に発生します。

1. **WAFv2 (Web ACL) のブロックログ確認**:
   * 特定のIPアドレスからのリクエストが WAF のルール（Rate Limit等）に抵触し、ブロックされているか確認。
2. **一次対応**:
   * 誤検知（False Positive）によるブロックの場合 ➔ WAF の該当ルールを一時的にカウントモード (Count) に変更。
   * 悪意ある攻撃 (DDoS) の場合 ➔ 攻撃元 IP を WAF のカスタムブロックリストに追加。

### C. `504 Gateway Timeout` (タイムアウト / DB・外部接続詰まり)
ECS アプリケーションからデータベース（Aurora）へのクエリ応答、または外部 API の応答がタイムアウト（デフォルト60秒）している場合に発生します。

1. **RDS Proxy ＆ データベース負荷の確認**:
   * RDS Proxy のアクティブ接続数、および Aurora の CPU 使用率を確認。
   * ロック競合や、インデックス未設定の重いクエリ（スロークエリ）がデータベースを占有していないか確認。
2. **一次対応**:
   * スロークエリの特定と強制終了 ➔ RDS コンソールより、データベースを圧迫しているプロセスIDを特定して `KILL` コマンドを実行。
   * コネクションプール枯渇の場合 ➔ ECS タスク数を一時的に手動でスケールアウトし、かつ RDS Proxy の最大接続制限を緩和。

---

## 3. インシデントの報告・エスカレーション

復旧作業が 15分 以内に完了しない、あるいは原因が特定できない場合は、速やかに SRE チャネル (`#sre-incident`) に以下の情報を報告してください。

* 発生時刻 (UTC / JST)
* エラーが発生しているエンドポイント
* 影響ユーザー数（概算）
* 直前に実施されたデプロイ・変更の有無
