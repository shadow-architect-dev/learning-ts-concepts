# SLO/SLI 運用および報告手順書 (SLO/SLI Reporting & Budget Policy Guide)

本ドキュメントでは、本アーキテクチャにおける SLO (Service Level Objectives) および SLI (Service Level Indicators) の継続的な計測、定例会議での報告フロー、ならびにエラーバジェットが枯渇した際の「エラーバジェットポリシー」の実行手順を定義します。

---

## 1. SLO/SLI の定義と計測方法

本システムでは、ユーザー体験に直結する以下の2つの指標を Datadog SLO 機能を用いて常時自動計測しています。

### ① 可用性 (Availability SLI)
- **指標 (SLI)**: ALB で正常に処理されたリクエストの割合（HTTPステータスが 5xx 以外のリクエスト数 / 全リクエスト数）
- **目標値 (SLO)**: **99.9%** (月間の許容ダウンタイム/エラー時間: 約43.2分)
- **Datadog での定義例**:
  - Service: `alb`
  - Query: `sum:aws.applicationelb.httpcode_elb_5xx{...}.as_count() / sum:aws.applicationelb.request_count{...}.as_count()`

### ② レイテンシ (Latency SLI)
- **指標 (SLI)**: ALB のターゲット応答時間が 500ms 未満で処理されたリクエストの割合（500ms未満のリクエスト数 / 全リクエスト数）
- **目標値 (SLO)**: **95.0%**
- **Datadog での定義例**:
  - Service: `alb`
  - Query: `pct95:aws.applicationelb.target_response_time.average{...} < 0.5`

## 2. 月次 SLO 報告書の作成手順 (毎月第1営業日実行)

SRE 担当者（または運用担当者）は、以下の手順に沿って月次の SLO 達成状況をドキュメント化し、定例会議に提出します。

### Step 1: Datadog からの実績値の抽出
1. Datadog コンソールにログインし、**[UX/SLO]** 画面を開きます。
2. 当月の対象期間（例: `30d` または `Month to Date`）を選択します。
3. 可用性およびレイテンシの **「実績値（Current Status）」** と **「エラーバジェット残量（Remaining Budget）」** の数値を控えます。

### Step 2: 報告書ファイルの作成
1. 本リポジトリの `docs/sre/reports/` ディレクトリ配下に、新規ファイルを作成します。
   - 命名規則: `YYYY-MM-report.md` (例: `2026-06-report.md`)
2. 本ドキュメントの末尾にある **「[4. 月次運用報告書 テンプレート]」** の内容をコピーし、新規作成したファイルに貼り付けます。
3. 控えた実績値とステータス（達成 🟢 / 未達 🔴）をテーブルに記入します。

### Step 3: エラーバジェット消費イベントの分析
当月中にエラーバジェットの急激な消費（バーンレートの急増）が発生していた場合、原因となったイベント（デプロイ、障害、トラフィックバースト等）を特定し、対策アクションを明記します。必要に応じて「障害振り返り報告書（Post-Mortem）」を起票し、リンクを記載します。

## 3. エラーバジェットポリシー (Error Budget Policy)

エラーバジェットの残量に応じて、開発チームと運用チームがとるべき強制アクションを定義します。これは感情的な対立を避け、システムの信頼性を組織的に維持するためのルールです。

| 補足バジェット残量 | ステータス | アラート ＆ アクション |
| :--- | :--- | :--- |
| **> 50% 残り** | 🟢 健全 | 通常運用。新機能デプロイの制限はありません。 |
| **20% 〜 50% 残り** | 🟡 警告 | Datadog の Burn Rate アラートをトリガーに、Slack 等で警告。信頼性向上タスクのバックログ化を行います。 |
| **0% 〜 20% 残り** | 🟠 危険 | 新機能のデプロイ時に追加のレビューを実施。開発チームに対して、次期スプリントでのリファクタリング枠の確保を打診します。 |
| **0% 未満 (枯渇)** | 🔴 制限 | **新機能のリリース（デプロイ）を原則一時停止。** 開発工数の50%以上を信頼性向上（バグ修正、スケール対策、クエリ最適化等）に強制的に割り当てます。 |

### リリース制限の解除基準
エラーバジェットが枯渇した場合、以下のいずれかを満たすまで新規リリースの停止を継続します。
1. 次の集計ウィンドウ（月次の切り替わり）でエラーバジェットが回復（プラスに転入）すること。
2. 障害の根本原因に対する恒久対策（ホットフィックス等）が本番環境へ適用され、かつ24時間のヘルスチェック期間を経て安定稼働が確認できること。
